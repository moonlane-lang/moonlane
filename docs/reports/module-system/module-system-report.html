This report is a full technical account of the Metel module system — what was implemented, how each pass works, what is still missing, and what technical debt exists. It is written to enable informed design decisions without requiring a code-reading session.

The module system is built as five strictly-ordered passes. Each pass produces a typed output that is the only accepted input to the next — there are no shortcuts or bypass paths. Every .mln file is a module; the module graph is discovered purely through import declarations.

Single-module compatibility: A .mln file with no import or export declarations is a complete program. Existing single-file programs remain valid without modification.

How modules map to files

:: in import paths maps directly to / in the filesystem:

Import declarationFile resolved
import parser::Ast;parser.mln
import parser::ast::Ast;parser/ast.mln
import root::a::b::c::T;a/b/c.mln (from root)

Pass 1 — Module Loading

module_loader.rs · 255 lines

Discovers and parses every .mln file reachable from the root via import declarations using depth-first recursion.

Cycle Detection

Uses a Vec<PathBuf> call stack. On a second visit to any path already in the stack, the full circular chain is reported as an error.

Deduplication

A HashSet<PathBuf> visited set ensures each file is loaded at most once, even if imported by many modules.

Data produced per file

pub struct LoadedModule {
    pub module_path: Vec<String>,  // e.g. ["parser", "ast"]
    pub file_path:   PathBuf,
    pub program:     Program,      // untyped AST, not yet name-resolved
}

pub struct ModuleGraph {
    pub root: PathBuf,
    pub modules: Vec<LoadedModule>,  // DFS order = topological order
}

Path root resolution

PathRootResolved to
root::[] — project root
self::current module's path
super::current module's path minus last segment
Name(n)current module's path + [n]
std:: imports return None from the resolver — they are virtual modules with no file and are silently skipped during loading.

The output Vec<LoadedModule> is in DFS traversal order: dependencies come before dependents. This ordering is relied on by all downstream passes.

Pass 2 — Name Resolution

name_resolver.rs · 722 lines

Builds the scope metadata used by every subsequent pass: which names each module exports publicly, and what imports each module can see.

pub struct ResolvedNames {
    pub scopes:      HashMap<Vec<String>, ModuleScope>,
    pub pub_surface: HashMap<Vec<String>, HashSet<String>>,
}

Three-pass structure inside Pass 2

2a
Collect pub names

Scan each module's AST for top-level declarations with the pub keyword. Populate pub_surface.

2b
Process export declarations

Re-exports extend pub_surface: export path::Name pulls a name from a submodule into this module's public API. Re-exporting a private item is error T0009.

2c
Resolve import declarations

Using the finalized pub_surface, record each import as an explicit ImportBinding or a glob entry. Then inject std::core into every module's globs at GlobTier::Std.

Pass 3 — Path Normalization

path_normalizer.rs · 341 lines

Rewrites every module-qualified expression path in the AST before the typechecker sees it. The typechecker only handles single-segment identifiers and ResolvedPath nodes — it must never receive a multi-segment path like std::core::Perhaps::Some.

Expression paths

Every Expr::Path with more than one segment is rewritten to Expr::ResolvedPath { resolved, original, span } where resolved is the local alias name.

Struct literal paths

std::core::Perhaps::Some { value: x } is rewritten to ["Perhaps", "Some"] — the module prefix is stripped.

The newtype guarantee

pub struct NormalizedModuleGraph(pub(crate) ModuleGraph);

// check_graph only accepts NormalizedModuleGraph —
// normalization cannot be skipped at compile time
Not validated: The normalizer does not check that referenced names are public. Paths it cannot resolve are left unchanged — the typechecker will emit T0003 or T0009.

Pass 4 — Typechecking

typechecker/mod.rs · 512 lines + inference.rs · construction.rs · registry.rs

check_graph processes modules in topological order, maintaining a GlobalExports accumulator that grows as each module is checked. This is how type information flows from dependency modules to importers. 

struct GlobalExports {
    modules: HashMap<ModulePath, ModuleExports>,
}
struct ModuleExports {
    pub_schemes: SchemeEnv,   // name → TypeScheme for every pub name
}

std::core is pre-seeded into GlobalExports before any module is processed, providing all built-in types to the entire graph.

Six steps per module

Step 1 — check_pub_annotations (lines 111–155)

Every pub fun must have explicit return type and parameter type annotations. Fails with T0010 if any annotation is missing. Runs before inference — unannotated pub functions cannot safely export a type scheme.

Step 2 — build_import_schemes (lines 252–346)

Builds a flat SchemeEnv of names this module is allowed to see. Processes globs in tier order (Std → User), then explicit imports. Fires T0009 (private name) or T0003 (absent name) on bad imports.

Step 3 — check_impl (lines 442–512)

Runs Hindley–Milner inference and typed AST construction. Imported schemes must be seeded into both the inference pass and the construction pass — this is the most critical invariant in the module system.

Critical invariant: ctx.bind_poly(name, scheme) AND scheme_env.entry(name).or_insert(scheme) must both be called for each imported name. Missing either produces silently wrong typed AST.
Step 4 — filter_pub_schemes (lines 396–425)

Filters the module's full scheme_env to only publicly accessible names: declared pub names plus names re-exported via export declarations.

Step 5 — GlobalExports.insert

Adds filtered pub_schemes to the accumulator. From this point, subsequent modules in topological order can import from this module.

Step 6 — imported_names population (lines 201–232)

Extracts the evaluator's cross-module seeding table. std::core glob entries are skipped — those come from register_builtins in the evaluator.

TypedModule output structure

pub struct TypedModule {
    pub module_path:    Vec<String>,
    pub decls:          Vec<TypedDecl>,
    pub import_aliases: HashMap<String, String>,               // alias → canonical
    pub imported_names: HashMap<String, (Vec<String>, String)>, // local → (module, canonical)
}

Pass 5 — Evaluation

evaluator/mod.rs · 900+ lines · evaluate_graph at lines 231–266

Processes modules in the same topological order as the typechecker. Builds one isolated Environment per module and stores them in an accumulator keyed by module path.

Cross-module mutual recursion (#189)

run_passes processes all three sub-passes for one module before moving to the next. This means if function foo in module A needs to call function bar in module B, and bar calls foo back, the pattern fails — when A is evaluated, B doesn't exist in module_envs yet.

Fix: Run Pass 1a for all modules before running Pass 1b for any module.

? operator as dedicated AST node (#214)

Expr::PropagateError is a special AST node with a mid-inference ctx.solve() call — the only place in the entire codebase where a partial solve occurs before constraint collection is complete. Desugaring to a match expression is planned but blocked until the From coercion edge case is addressed.

Technical Debt

High

Dual path computation (module_loader vs. name_resolver)

Two independent functions implement the same "resolve PathRoot → absolute path" logic. If they diverge, GlobalExports lookups fail silently with T0009/T0003 errors on valid imports. No shared implementation, no test asserting agreement. Fix: extract into a single resolve_path_root in src/module_paths.rs.

High

Dual-registration invariant unenforced

Imported schemes must be seeded into both the inference pass and the construction pass. This is documented in ADR-0022 but enforced only by code convention — there is no type-level guarantee and no test that exercises a missing-one-side case. Fix: make InferContext::new accept &SchemeEnv as a required parameter.

High

evaluator.md Known Limitations are stale

The Known Limitations section still documents two problems fixed in sprint 13 (flat module environment, declaration name collisions). Anyone reading it believes the module system is broken in ways it no longer is.

Medium

StdPrelude / register_builtins can silently diverge

StdPrelude::schemes() (typechecker) and register_builtins (evaluator) are separately maintained. If someone adds a name to one but not the other, the mismatch either passes typechecking but fails at runtime, or is available at runtime but rejected by the typechecker. Fix: add a test asserting both key sets are equal.

Medium

Mid-inference partial solve in PropagateError

inference.rs:669–714 calls ctx.solve() mid-way through type inference — the only such site in the codebase. It can interact with constraints added after the ? expression in non-obvious ways. The current implementation also silently ships partial From coercion support that was never explicitly planned.

Low

T0009 detection scans raw AST (quadratic + back-channel)

When an explicit import is not found in GlobalExports, build_import_schemes scans the source module's entire raw program.decls to decide between T0009 (private) and T0003 (absent). This is O(n×m) and accesses pre-normalization AST while the typechecker is supposed to operate only on NormalizedModuleGraph.

Low

type_context accumulator is approximate

Struct/enum/aspect declarations from already-checked modules are accumulated as raw untyped AST and re-processed from scratch for each module. If a struct in module A references a type from module C, and module B hasn't imported C, building B's type registry may fail. Currently untriggered by tests, but latent for complex multi-module programs.
