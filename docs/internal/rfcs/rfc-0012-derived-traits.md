## Summary

Define the language's meta-feature layer: attributes, metadata annotations, and macros — and within that layer, the mechanism for automatically generating trait implementations (derive). These are grouped because derive cannot be designed in isolation: it is either a closed compiler-known feature (no meta-feature layer needed), or it is the first user of a general attribute or macro system. The choice between those two paths is the central open question. All features in this RFC are tentatively deferred to **v0.5+**, after the core language (generics, traits, concurrency, memory model) is stable.

---

## Motivation

### Derived trait implementations

Writing `impl Eq for Point { fun eq(self, other: Point) -> Bool { self.x == other.x && self.y == other.y } }` by hand for every struct is tedious and error-prone. A derive mechanism generates these implementations structurally — field-by-field for structs, variant-by-variant for enums. The primary use cases are `Eq`, `Ord`/`Comparable`, `Display`, `Clone`, `Hash`, and `Linear` (for types that are linear by structure rather than explicit declaration).

### Attributes and metadata

Beyond derive, a general attribute/metadata system enables:
- Compiler hints (`#[inline]`, `#[cold]`, `#[must_use]`)
- Conditional compilation (`#[cfg(...)]`)
- FFI annotations (`#[extern("C")]`)
- Lints and suppressions (`#[allow(...)]`, `#[deny(...)]`)
- Documentation metadata (`#[doc(...)]`)

Without a principled attribute syntax, these accumulate as ad-hoc keywords or magic comments. A single syntax form (`@`) handles all of them uniformly.

### Macros

A macro system enables syntactic abstraction — generating code from a compact notation. Whether Gust wants macros at all is an explicit open question. If derive is implemented as a macro, the macro system is the prior requirement. If derive is a compiler-built-in, macros remain optional.

## Preferred Syntax: `@`

The preferred grammar symbol for attributes and metadata is `@`. This is distinct from all current Gust operators and consistent with annotation syntax in several modern languages (Java, Python decorators, Zig's `@builtins`).

```gust
@derive(Eq, Ord, Display)
struct Point {
    x: Float,
    y: Float,
}

@inline
fun fast_path(n: Int) -> Int { n * 2 }

@cfg(target = "linux")
fun platform_init() { ... }

@allow(unused)
let _debug_value = compute();
```

Multiple attributes stack vertically, one per line, before the item they annotate. Attributes apply to the next declaration or binding — they do not apply to expressions.

The `@` form is preferred over Rust's `#[...]` because:
- `#` is visually associated with comments in many languages; `@` is unambiguously an annotation sigil
- `@` is already unused in Gust's grammar
- `@(...)` is unambiguous as a prefix — no bracket/brace confusion with other constructs

## Proposal Space

Three implementation paths exist for derive specifically. The RFC does not choose between them yet — that decision is deferred.

### Path A — Compiler-built-in derive (no macro system)

Derive is a closed set of structurally derivable traits known to the compiler. The `@derive(...)` attribute is handled directly by the typechecker/codegen with no user extensibility.

```gust
@derive(Eq, Ord, Display)
struct Point { x: Float, y: Float }
```

**Pros:** simple to implement, no macro infrastructure needed, guaranteed correct implementations.  
**Cons:** user-defined derivable traits are impossible. Third-party libraries cannot add derivable traits.

### Path B — Attribute macros (procedural macros)

`@derive(Trait)` expands to an `impl` block generated by a macro associated with `Trait`. User-defined traits can be made derivable by providing a derive macro. This is Rust's model.

```gust
// Library author provides:
macro derive_eq for Eq { ... generates impl Eq ... }

// User writes:
@derive(Eq)
struct Point { x: Float, y: Float }
```

**Pros:** fully extensible. **Cons:** macro system is a major language feature; procedural macros in Rust are notoriously complex to write and maintain.

### Path C — Derive as a language keyword (no attribute syntax needed for this case)

Derive is expressed with a keyword rather than the `@` attribute system:

```gust
struct Point derives Eq, Ord, Display {
    x: Float,
    y: Float,
}
```

The `derives` keyword is closed (compiler-known) but readable and requires no attribute infrastructure. The `@` system can still be introduced separately for other metadata purposes.

**Pros:** derive is ergonomic and self-contained; attribute syntax is not a prerequisite. **Cons:** `derives` is a new keyword; does not scale to user-extensible derivation.

## Derivable Traits (initial set)

Regardless of implementation path, the initial set of structurally derivable traits:

| Trait | Behaviour |
|---|---|
| `Eq` | Field-by-field equality |
| `Ord` / `Comparable` | Lexicographic field ordering |
| `Display` | Structural `to_string` (see open question below) |
| `Clone` | Deep field-by-field clone |
| `Hash` | Structural hash combining all fields |
| `Linear` | Marks the type linear if all fields are linear (see RFC-0024) |

`Linear` as a derivable trait is an alternative to the `linear` keyword on the declaration — a type with all-linear fields could derive rather than declare linearity, making the structural rule explicit.

## Interaction with Other RFCs

### RFC-0024 (Linear Types)

The `linear` keyword on struct/enum declarations could be replaced by `@derive(Linear)` or `struct Foo derives Linear`. Both express the same thing. The keyword form is simpler; the derive form is more uniform. This is an open question for RFC-0024's final form.

### RFC-0001 (Pointers) and RFC-0026 (Unsafe Blocks)

`@extern("C")` for FFI function signatures (RFC-0026 open question 4) would use the `@` attribute syntax defined here. The attribute system is a soft prerequisite for a clean FFI story.

### RFC-0009 (Module System)

`@pub`, `@cfg`, and documentation attributes interact with the module system's visibility model. The attribute syntax should be designed with module-level concerns in mind.

### RFC-0011 (Operator Overloading)

Deriving `Eq` and `Ord` generates implementations of the operator traits. RFC-0011 must be accepted before derived `Eq`/`Ord` can be implemented.

## Alternatives Considered

### No attribute syntax — ad-hoc keywords only

Each compiler directive is its own keyword or syntax form. This avoids designing a general system but leads to keyword proliferation and inconsistency. Rejected as a long-term position; acceptable only before v0.5 when no attribute-dependent features have shipped.

### `#[...]` Rust-style attributes

Familiar to Rust programmers but visually ambiguous with comments (`#`). Rejected in favour of `@`.

### Lisp-style macros (hygienic, syntax-level)

Full hygienic macro system allowing arbitrary syntactic transformation. Maximum power, maximum complexity. Well outside the scope of a v0.5 feature for a language at v0.1. Not ruled out as a distant future direction.

## Open Questions

1. **Path choice.** Compiler-built-in (Path A), procedural macros (Path B), or `derives` keyword (Path C)? Path A and C are compatible — Path C for derive, `@` for everything else. Path B requires the macro system first.

2. **`@` attribute scope.** What items can be annotated — struct/enum declarations, function declarations, `let` bindings, individual fields? Field-level attributes (e.g. `@skip` on a field to exclude it from `Display`) are useful but add parsing complexity.

3. **`Display` vs `From` for string conversion.** `print` currently only accepts `String`. When traits land, `print` should accept any type with a string representation. The question is which trait owns that conversion:
   - A `Display` trait (`fun to_string(self) -> String`) implemented by the source type — the natural direction for user-defined types.
   - `String` implementing `From<T>` for each printable type — consistent with the `from` pattern but puts the responsibility on `String`, which cannot know about user-defined types without open dispatch.
   These serve different purposes and should likely remain separate traits. Resolve before finalising the `print` signature.

4. **Compiler-known attribute registry.** Even with Path A or C, the compiler needs a fixed set of recognised `@` attributes (e.g. `@inline`, `@cfg`, `@allow`). Should unknown `@` attributes be a compile error, a warning, or silently ignored (for forward compatibility)?

5. **`@cfg` and conditional compilation.** Conditional compilation is a significant feature in its own right (platform-specific code, feature flags). Should `@cfg` be in scope for this RFC or a separate one?

6. **`linear` keyword vs `@derive(Linear)`.** Should RFC-0024's `linear` keyword be removed in favour of `@derive(Linear)` or `derives Linear` once this RFC is accepted? The keyword form is available sooner (v0.3); the derive form is more uniform but requires v0.5+. A possible migration: accept `linear` keyword now, deprecate in favour of derive when the meta-feature layer lands.

---

## Timing Recommendation

All features in this RFC are **deferred to v0.5+**. The core language must stabilise first:

- Generics and traits (v0.2) — derive requires traits to exist
- Memory model (v0.3) — `@extern` for FFI requires unsafe blocks (RFC-0026)
- Concurrency (v0.4) — `@cfg` and platform attributes interact with the concurrency model

The `linear` keyword (RFC-0024) is the one meta-adjacent feature that ships early (v0.3) as a plain keyword, to be revisited for derive integration at v0.5.

Minimum action before v0.5: reserve `@` as a grammar token so it cannot be used for other purposes. Reserve `derives` as a keyword for the same reason. This prevents a breaking change when the attribute system lands.
